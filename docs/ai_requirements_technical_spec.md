# Технические требования к веб-платформе управления требованиями с ИИ

## 1. Архитектурное видение
- **Цель:** изолированная веб-платформа, реализующая алгоритм управления требованиями и изменениями, запускаемая локально с минимальной стартовой сложностью.
- **Принцип MVP:** однопользовательское использование без ролевой модели; архитектура должна допускать дальнейшее расширение без полной переработки.
- **Слоистая архитектура:**
  - Презентационный слой: лёгкий веб-интерфейс на базе шаблонов.
  - Сервисный слой: REST-бэкенд FastAPI со встроенным управлением состояниями.
  - Слой интеллектуальных сервисов: ИИ-модули A–E (локально или через облачные API).
  - Слой данных: реляционная БД, локальное файловое хранилище, векторный поиск.
  - Системный слой: простая аутентификация и локальный деплой.

## 2. Фронтенд
- **Технологический стек:** одностраничное приложение на базе Svelte + Vite с Tailwind CSS и компонентной библиотекой Flowbite-Svelte. В качестве стартовой точки используется готовый шаблон **Flowbite-Svelte Admin Dashboard**, адаптированный под доменные сущности (Expectations, Requirements, Change Requests). Такой подход обеспечивает лёгкость стека и одновременно даёт готовую навигацию, таблицы, модальные окна и форму чата без необходимости разрабатывать layout с нуля. Low-code платформы не используются.
- **Структура экранов:**
  - `Workspace`: основная панель на базе admin-dashboard layout (sidebar + top bar) с навигацией по разделам (Expectations, Requirements, Change Requests, Archive, Settings).
  - `Intake`: мастер загрузки ожиданий с текстовой формой и загрузкой файлов (transcript `.txt/.md/.pdf`).
  - `Requirement Studio`: редактирование требований с вкладками One-Pager, User Story, Acceptance Criteria, Metrics и встроенным редактором (TipTap/ProseMirror на базе Svelte).
  - `AI Chat`: боковая панель/модальное окно, встроенная в правую колонку шаблона dashboard, с историей обращений к LLM для уточнения ожиданий и генерации артефактов.
  - `Search & Filters`: полнотекстовый поиск с NLQ, фильтрация по статусам и метрикам, переключение статусов из списка.
- **Компоненты и взаимодействия:**
  - Карточки сущностей с быстрыми действиями (edit, approve, send to change control, archive).
  - Динамические формы для конфигурируемых метрик (добавление/удаление полей, выбор типа значения).
  - Таблица версий с просмотром диффов и кнопкой отката.
  - Notifications/toasts для обратной связи об операциях и статусах фоновых задач.
- **UI/UX требования:**
  - Интерфейс на английском языке, светлая тема по умолчанию, базовая адаптивность (desktop-first, tablet fallback).
  - Использование готовых UI-шаблонов Tailwind/Tailwind UI для единообразия и быстрого запуска.
  - Горячие клавиши для навигации (например, `Ctrl+K` для глобального поиска).
- **Интеграция с ИИ:**
  - Все обращения к LLM выполняются через REST эндпоинты бэкенда.
  - Чатовый интерфейс хранит контекст диалога в состоянии фронтенда и при необходимости передаёт историю сообщений в API.

## 3. Бэкенд
- **Стек:** Python 3.11 + FastAPI для быстрой реализации REST API.
- **API:**
  - REST endpoints для CRUD-операций над сущностями `Expectation`, `Requirement`, `ChangeRequest`, `Decision`, `Version`, `MetricDefinition`, `MetricValue`.
  - REST endpoints для запуска ИИ-пайплайнов (извлечение ожиданий, генерация требований, анализ зависимостей и конфликтов, impact-анализ, NLQ-поиск).
  - WebSockets и Server-Sent Events не используются; прогресс долгих операций отображается опросом.
- **Workflow:** управление жизненным циклом реализовано в FastAPI через лёгкую state-machine библиотеку (например, `transitions`) либо собственную реализацию.
- **Безопасность:** минимальная аутентификация (единый логин/пароль или токен); роли и сложные политики отсутствуют.
- **Валидация:** Pydantic-модели обеспечивают проверку входных данных и поддержку гибких схем метрик.
- **Фоновые задачи:** запускаются только для длительных ИИ-операций (FastAPI BackgroundTasks либо отдельный воркер на Celery/RQ); при возможности операции выполняются синхронно без очереди.

## 4. ИИ-слой
- **Оркестратор:** модуль в составе бэкенда или отдельный процесс, управляющий обращениями к языковым моделям и векторному поиску.
- **Модуль A (Extractor & Classifier):** извлечение ожиданий из текста/файлов, нормализация формулировок, классификация по области/типу/ролям, расчёт impact-score и ICE, выявление дублей и конфликтов.
- **Модуль B (Generator):** генерация One-Pager, User Story, Acceptance Criteria (Gherkin), предложений по RACI.
- **Модуль C (Dependency & Conflict Analyzer):** анализ зависимостей между требованиями и существующими элементами, выявление конфликтов.
- **Модуль D (Impact Analysis):** оценка влияния Change Request на требования, роли, риски, версии.
- **Модуль E (NLQ Search):** обработка поисковых запросов и рекомендации по созданию новых ожиданий или CR.
- **Исполнение:** поддерживаются локальные LLM (например, LAMU) и внешние API (OpenAI GPT-5 при доступности); переключение режимов выполняется через конфигурацию без изменения контрактов.

## 5. Хранилище данных
- **Реляционная БД:** PostgreSQL 14+ (предпочтительно для поддержки PGVector) либо SQLite для полностью локальной поставки.
  - Основные таблицы: `Expectation`, `Requirement`, `RequirementVersion`, `ChangeRequest`, `ChangeRequestVersion`, `Decision`, `MetricDefinition`, `MetricValue`, `Attachment`.
  - Динамические атрибуты и наборы метрик хранятся в JSON/JSONB-полях.
- **Файловое хранилище:** локальная директория (например, `./storage`) для транскриптов и других артефактов; доступ организован через API бэкенда.
- **Векторное хранилище:** PGVector внутри PostgreSQL или локальный FAISS-индекс для семантического поиска и дедупликации.
- **Кэширование:** не используется; все операции выполняются через основную БД.

## 6. Управление метриками
- **Конфигурируемость:**
  - Таблица `MetricDefinition` хранит параметры метрик (название, описание, тип данных, единицы, правила агрегации).
  - Таблица `MetricValue` связывает сущность с метрикой и значением, фиксируя источник (ИИ или пользователь).
  - UI позволяет добавлять и изменять метрики без перезапуска системы; количество метрик не ограничено.
- **Расчёты:**
  - Поддерживаются пользовательские формулы (простые выражения) в песочнице.
  - ИИ может предлагать значения метрик на основе анализа контента ожиданий и требований.

## 7. Деплоймент и инфраструктура
- **Запуск:** все компоненты работают локально; предпочтительно использовать `docker-compose` с сервисами `frontend`, `backend`, `postgres` (или `sqlite` без контейнера), `vector-store` (при отдельном сервисе), `ai-worker` (если фоновые задачи вынесены).
- **CI/CD:** локальные скрипты/Makefile; допускается подключение GitHub Actions для автоматических проверок и сборки контейнеров при условии возможности локального запуска.
- **Мониторинг и логирование:** специализированные стеки (Grafana, Promtail, ELK) не применяются; достаточно стандартного вывода логов приложений.
- **Резервное копирование:** ручные или скриптовые локальные бэкапы БД и файлового хранилища.

## 8. Безопасность и эксплуатация
- **Сеть:** доступ по HTTP (без HTTPS) в рамках локального окружения.
- **Аутентификация:** упрощённая; все пользователи обладают полным доступом.
- **Логирование:** подробные структурированные логи и аудит действий не ведутся.
- **Fallback-сценарии:** не реализуются; ошибки отображаются пользователю напрямую.
- **Эксплуатация:** предполагается локальный запуск с возможностью переноса на сервер в будущем без изменения архитектурных решений.

## 9. План расширения после MVP
- Добавление ролевой модели и мультиарендности.
- Интеграция с внешними системами (таск-трекеры, каталоги пользователей).
- Внедрение HTTPS, продвинутой безопасности и мониторинга.
- Расширенная аналитика, аудит и автоматические fallback-сценарии.

## 10. Критерии готовности технической платформы
- Фронтенд, бэкенд, ИИ-слой и хранилище разворачиваются локально одной командой (`docker-compose up` либо эквивалент).
- ИИ-модули доступны через единый REST API и переключаются между локальным и облачным режимами настройкой конфигурации.
- Пользователь может пройти весь цикл: ввод ожиданий, генерация требований, анализ зависимостей, согласование, управление изменениями, архивация.
- Система поддерживает поиск, версионирование и хранение истории решений в одной базе данных без внешних сервисов.
