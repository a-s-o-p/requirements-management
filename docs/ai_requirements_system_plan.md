# План разработки веб-платформы управления требованиями с ИИ

## 1. Видение продукта и ограничения MVP
- **Назначение:** автономная веб-платформа, реализующая алгоритм управления требованиями и изменениями, описанный в методике, без зависимости от внешних инструментов.
- **Первая версия:** однопользовательское (single-tenant) решение с простой аутентификацией и минимальным профилем (имя, email). Расширяемость до много-пользовательского режима закладывается архитектурно, но не реализуется в MVP.
- **Скорость вывода:** максимальное упрощение ручных операций, перенос сложности на ИИ-модули и конфигурируемые справочники, чтобы уложиться в кратчайшие сроки поставки.

## 2. Роли и сценарии
- **Главный пользователь (PM/BA/PO):** создаёт ожидания, принимает решения по требованиям, управляет статусами.
- **Исполнитель / QA (опционально в MVP):** получает назначенные требования, отмечает выполнение по AC.
- **ИИ-службы (A–E):** автоматически выполняют извлечение, генерацию и анализ, сокращая ручные шаги.

## 3. Общие функциональные требования
- Веб-интерфейс с навигацией по стадиям: Ожидания → Требования → В работе → Проверка → Архив; отдельная вкладка «Change Requests».
- Поиск по ожиданиям, требованиям и архиву (NLQ + фильтры).
- Управление статусами согласно жизненному циклу: Expectation → Verified → Requirement Draft → DoR Ready → Approved → In Work → V&V → Done → Archived.
- Создание и обработка Change Request с привязкой к версиям требований.
- Гибкое управление метриками и оценками (Impact, ICE, Success Metrics и др.) с возможностью добавлять/удалять показатели без изменения кода.

## 4. Модуль A — Инициация ожиданий
- **Входные каналы MVP:**
  - Ручная форма ввода текста ожидания.
  - Загрузка файла (текстовая транскрипция встречи). Допустимый размер и формат задаются конфигурацией.
- **ИИ-пайплайн:**
  - Нормализация текста, извлечение множества ожиданий из одного источника.
  - Классификация по области, типу (Business/Functional/NFR), вовлечённым ролям.
  - Расчёт Impact-score, ICE и рекомендации по MoSCoW с возможностью переопределения человеком.
  - Обнаружение дублей/конфликтов внутри текущей партии и с архивом.
- **UI:** мастер импорта с предпросмотром обнаруженных ожиданий, тегами и оценками.

## 5. Модуль B — Верификация ожиданий
- Панель triage: списки на подтверждение/уточнение/отклонение с комментариями.
- Возможность группового утверждения и корректировки MoSCoW/ICE.
- Лог действий сохраняется для аудита.

## 6. Модуль C — Формирование требований
- **ИИ-генератор:**
  - Создание One-Pager (Problem, Goal, Value, Success Metrics, Scope, Assumptions, Risks).
  - Генерация User Story и Acceptance Criteria (Gherkin).
  - Предложение RACI (с фиксацией ответственных по имени/email).
- **Гибкие метрики:**
  - Хранилище настроек метрик позволяет добавлять новые показатели (например, NPS, ROI) и привязывать их к требованиям.
  - Интерфейс выбора метрик (checklist + пользовательские поля).
- **Редактор:** визуальный просмотр черновика, правка текста, связь «несколько ожиданий ↔ несколько требований».

## 7. Модуль D — Анализ зависимостей и DoR
- Построение графа связей по сущностям: системы, процессы, роли, артефакты.
- ИИ-модуль выявляет конфликты и блокеры; результаты отображаются в карточке требования.
- Чеклист Definition of Ready: ≥3 AC, закрытые блокеры/конфликты или план их снятия, подтверждённые метрики.

## 8. Модуль E — Согласование и утверждение
- Автоматизированный workflow по RACI: отправка уведомлений (в MVP — email/интерфейсное уведомление) и фиксация ответов.
- Decision Log с указанием роли, решения, комментария и срока.
- Только требования со статусом DoR Ready и Approved доступны для передачи в работу.

## 9. Модуль F — Передача в работу и исполнение
- Назначение ответственного исполнителя и окна работ.
- Фиксация Success Metrics и ответственного за проверку AC.
- Перевод в статус In Work.

## 10. Модуль G — Верификация исполнения (V&V)
- Чеклист Acceptance Criteria с отметками выполнения и возможностью прикрепить артефакты.
- ИИ-модуль протоколирует результаты (вытягивает ключевые выводы, формирует резюме проверки).
- Перевод в статус Done или возврат на доработку.

## 11. Модуль H — Управление изменениями
- Поток Change Request: создание (связь с требованием), triage, Impact Analysis (ИИ-модуль D), согласование, обновление SemVer.
- Возможность сравнения версий требования (diff по тексту One-Pager, User Story, AC).

## 12. Модуль I — Архив и поиск
- Хранение всех версий требований, историй решений, связей Expectation → Requirement → V&V → Version.
- NLQ-поиск (ИИ-модуль E) + фильтры по статусу, типу, метрикам.
- При отсутствии точного совпадения система предлагает создать новое ожидание и уведомляет ответственных.

## 13. Конфигурируемость метрик и оценок
- Таблица `MetricDefinition` для описания метрик (название, тип данных, единицы, метод расчёта).
- Связующая таблица `MetricValue` с возможностью прикреплять несколько метрик к ожиданию/требованию.
- Настройки порогов/формул задаются администратором через конфиг-интерфейс или YAML-файл.

## 14. Архитектурный скелет
- **Фронтенд:** SPA (React/Vue) с модульным UI-kit, поддержкой тёмной/светлой темы.
- **Бэкенд:** REST/GraphQL API, реализующий workflow и интеграции с ИИ-сервисами.
- **ИИ-слой:**
  - LLM-сервис для генерации/классификации (можно стартовать с облачной модели с возможностью замены на on-premise).
  - Аналитические пайплайны для Impact/ICE и дедупликации.
- **Хранилище:** реляционная БД (PostgreSQL) с поддержкой JSONB для гибких структур метрик.
- **Очередь задач:** для асинхронной обработки загрузок, генерации и анализов.
- **Развёртывание MVP:** контейнеризированное приложение (Docker Compose) в одном окружении.

## 15. UX-требования
- Чистый, интуитивно понятный интерфейс с карточками ожиданий/требований, цветовой индикацией статусов.
- Панель быстрого действия (создать ожидание, запустить анализ, подать CR).
- Хлебные крошки и шаги мастеров для длинных процессов (импорт ожиданий, согласование).
- Локализация интерфейса (RU/EN) закладывается структурно, но в MVP можно ограничиться одним языком.

## 16. Нефункциональные требования
- **Изоляция:** система не требует внешних интеграций; все данные управляются внутри платформы.
- **Безопасность:** базовая аутентификация, шифрование данных в покое и при передаче (HTTPS, at-rest encryption).
- **Надёжность:** журналирование событий, резервные копии БД.
- **Масштабируемость:** возможность вынести ИИ-сервисы и фронтенд/бэкенд в отдельные сервисы на следующем этапе.

## 17. Дорожная карта MVP (2 недели)
- **Неделя 1:** настройка бэкенда, моделей данных (Expectation, Requirement, ChangeRequest, Decision, Version, MetricDefinition, MetricValue), мастер импорта, первичный NLP-конвейер, генерация One-Pager/User Story/AC.
- **Неделя 2:** DoR чеклист, approval workflow, V&V панель, базовый Change Request, архив и NLQ-поиск, финальные полировки UI.

## 18. Критерии успешности MVP
- Полный проход сценария от ввода ожидания до архивации требования, включая смену статусов и фиксацию метрик.
- Работа всех ИИ-модулей A–E в автоматическом режиме с возможностью ручной коррекции.
- Веб-интерфейс обеспечивает создание, поиск, редактирование и контроль изменений без обращения к внешним системам.
